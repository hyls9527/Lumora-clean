## 4. 开发规范

### 4.1 分支策略

| 分支 | 用途 | 保护 |
|------|------|------|
| `main` | 生产分支 | ✓ |
| `develop` | 开发分支 | ✗ |
| `feature/*` | 功能分支 | ✗ |
| `fix/*` | 修复分支 | ✗ |

**代码依据**：`.github/CODEOWNERS`

---

### 4.2 提交规范

```
<type>(<scope>): <subject>

type: feat | fix | docs | style | refactor | test | chore
scope: 前端 | 后端 | 文档 | CI
subject: 简短描述
```

---

### 4.3 代码审查

- 所有 PR 需要至少 1 个审批
- CI 必须通过
- 测试覆盖率不能下降

**代码依据**：`.github/PULL_REQUEST_TEMPLATE.md`

---

## 5. 质量门禁

### 5.1 提交前检查

```bash
# 前端
npx tsc --noEmit      # TypeScript 类型检查
npx vitest run        # 前端测试

# 后端
cargo test --lib      # Rust 测试
cargo check           # Rust 编译检查
```

### 5.2 CI 检查

| 检查 | 工具 | 阈值 |
|------|------|------|
| TypeScript 类型 | tsc --noEmit | 0 errors |
| 前端测试 | vitest | 100% pass |
| Rust 测试 | cargo test | 100% pass |
| Rust 编译 | cargo check | 0 errors |
| 安全审计 | cargo audit | 0 critical |

**代码依据**：`.github/workflows/ci.yml`
