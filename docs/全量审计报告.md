## 2. 安全专家视角

### S-1. 手动 CASCADE DELETE — 🟡 中

**问题**：`trash.rs` 中手动执行多条 DELETE 语句来清理关联数据，而非使用数据库级 CASCADE。

```rust
// trash.rs:52-68
tx.execute("DELETE FROM image_tags WHERE image_id = ?1", ...)?;
tx.execute("DELETE FROM analysis_history WHERE image_id = ?1", ...)?;
tx.execute("DELETE FROM vec_embeddings WHERE image_id = ?1", ...)?;
tx.execute("DELETE FROM embeddings WHERE image_id = ?1", ...)?;
tx.execute("DELETE FROM images WHERE id = ?1", ...)?;
```

**影响**：如果新增表（如 `variant_groups`）关联 `images.id`，但忘记更新删除逻辑，会产生孤儿记录。

**修复复杂度**：中（需要修改 Schema 或确保删除逻辑完整）

---

### S-2. 无 SQL 注入风险 — ✅

所有 SQL 查询使用参数化查询（`params![]` 宏），无字符串拼接。

---

### S-3. CSP 配置合理 — ✅

```
connect-src 'self' http://localhost:* https://localhost:* https://github.com ipc: tauri:
```

仅允许 localhost 和 github.com（用于更新检查），无外部数据泄露风险。

---

### S-4. 路径遍历风险 — 🟢 低

`import_images` 接受用户输入的路径，但因为是本地桌面应用，用户本身就是文件系统的所有者，风险极低。

## 3. 性能专家视角

### P-1. Mutex 在 async 上下文中使用 — 🟡 中

**问题**：`ai.rs` 中在 async 函数内使用 `db.conn().lock()`。

```rust
// ai.rs:268
let conn = db.conn().lock().map_err(|_| AppError::Lock)?;
```

**影响**：如果锁持有时间长（如等待 Ollama API 响应），会阻塞其他数据库操作。

**实际情况**：检查代码发现，锁在数据库操作后立即释放，网络调用在锁外进行。**当前实现正确**。

**建议**：保持现状，但添加注释说明锁的范围。

---

### P-2. IPC 调用数量合理 — ✅

页面加载时仅发起 17 次 IPC 调用，无 flooding 风险。

---

### P-3. 无串行 await 问题 — ✅

未发现 `await...await` 的串行模式。

## 4. 数据完整性专家视角

### D-1. 手动删除逻辑完整性 — 🟡 中

**问题**：`permanent_delete_image` 手动删除 5 个表的记录，但如果新增表关联 `images.id`，需要手动更新。

**当前覆盖**：
- ✅ `image_tags`
- ✅ `analysis_history`
- ✅ `vec_embeddings`
- ✅ `embeddings`
- ✅ `images`
- ❓ `variant_groups`（通过 `variant_group_id` 关联，但删除图片不影响变体组）

**建议**：在 `permanent_delete_image` 函数顶部添加注释，列出所有需要清理的表。

迁移使用 `IF NOT EXISTS`，支持幂等执行。版本通过 `PRAGMA user_version` 追踪。
