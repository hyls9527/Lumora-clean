**Part A: Implement `db/connection.rs`**

Replace the stub with the full database initialization logic. This module is responsible for opening the SQLite database, configuring PRAGMAs, running migrations, and setting up FTS5 triggers.

Key functions:

1. **`open_database(db_path: &Path) -> Result<Connection, rusqlite::Error>`**
   - Open connection with `Connection::open(db_path)`
   - Execute batch PRAGMAs: `journal_mode=WAL`, `busy_timeout=5000`, `foreign_keys=ON`, `synchronous=NORMAL`
   - Return the connection

2. **`run_migrations(conn: &mut Connection) -> Result<(), Box<dyn std::error::Error>>`**  
   - Call `migrations::MIGRATIONS.to_latest(conn)?`
   - This is wrapped in rusqlite_migration's internal transaction — crash-safe per D-17

3. **`setup_fts5_triggers(conn: &Connection) -> Result<(), rusqlite::Error>`**
   - Execute the FTS5 trigger SQL from RESEARCH.md Pattern 2 (with the corrected column names matching the `images` table schema):
   
   AFTER INSERT trigger:
   ```sql
   CREATE TRIGGER IF NOT EXISTS images_fts_ai AFTER INSERT ON images
   BEGIN
       INSERT INTO images_fts(rowid, original_name, tags)
       VALUES (new.rowid, new.original_name,
           (SELECT GROUP_CONCAT(t.name, ' ') FROM image_tags it
            JOIN tags t ON t.name = it.tag_name
            WHERE it.image_id = new.id));
   END;
   ```
   
   AFTER DELETE trigger:
   ```sql
   CREATE TRIGGER IF NOT EXISTS images_fts_ad AFTER DELETE ON images
   BEGIN
       INSERT INTO images_fts(images_fts, rowid, original_name, tags)
       VALUES ('delete', old.rowid, old.original_name,
           (SELECT GROUP_CONCAT(t.name, ' ') FROM image_tags it
            JOIN tags t ON t.name = it.tag_name
            WHERE it.image_id = old.id));
   END;
   ```
   
   AFTER UPDATE trigger (using delete+insert pattern per RESEARCH.md Pitfall 1 to avoid FTS5 token leakage):
   ```sql
   CREATE TRIGGER IF NOT EXISTS images_fts_au AFTER UPDATE ON images
   BEGIN
       INSERT INTO images_fts(images_fts, rowid, original_name, tags)
       VALUES ('delete', old.rowid, old.original_name,
           (SELECT GROUP_CONCAT(t.name, ' ') FROM image_tags it
            JOIN tags t ON t.name = it.tag_name
            WHERE it.image_id = old.id));
       INSERT INTO images_fts(rowid, original_name, tags)
       VALUES (new.rowid, new.original_name,
           (SELECT GROUP_CONCAT(t.name, ' ') FROM image_tags it
            JOIN tags t ON t.name = it.tag_name
            WHERE it.image_id = new.id));
   END;
   ```

**Part B: Implement `db/search.rs` FTS5 search**

Replace the stub with FTS5 parameterized search using the `images_fts` virtual table. Per RESEARCH.md Pattern 2, use the join pattern:

```rust
pub fn search_images_fts(
    conn: &Connection,
    query: &str,
    limit: u32,
    offset: u32,
) -> Result<Vec<SearchResultRow>, rusqlite::Error> {
    let sql = "
        SELECT i.id, i.file_path, i.original_name, i.width, i.height, i.file_size,
               i.format, i.rating, i.favorite, i.created_at, i.thumbnail_path,
               COALESCE(
                   (SELECT GROUP_CONCAT(t.name, ' ') FROM image_tags it
                    JOIN tags t ON t.name = it.tag_name
                    WHERE it.image_id = i.id),
                   ''
               ) AS tags,
               fts.rank AS fts_rank
        FROM images i
        JOIN images_fts fts ON i.rowid = fts.rowid
        WHERE images_fts MATCH ?1
          AND i.is_deleted = 0
        ORDER BY fts.rank
        LIMIT ?2 OFFSET ?3
    ";
    
    let mut stmt = conn.prepare(sql)?;
    let rows = stmt.query_map(params![query, limit, offset], |row| {
        Ok(SearchResultRow {
            id: row.get("id")?,
            file_path: row.get("file_path")?,
            original_name: row.get("original_name")?,
            width: row.get("width")?,
            height: row.get("height")?,
            file_size: row.get("file_size")?,
            format: row.get("format")?,
            rating: row.get("rating")?,
            favorite: row.get::<_, i32>("favorite")? != 0,
            created_at: row.get("created_at")?,
            thumbnail_path: row.get("thumbnail_path")?,
            tags: row.get("tags")?,
            fts_rank: row.get("fts_rank")?,
        })
    })?;
    
    rows.collect()
}
```

**CRITICAL SQL injection prevention:** Use `params![query, limit, offset]` with `?1, ?2, ?3` bind parameters. NEVER use string interpolation for the FTS5 MATCH query — even though MATCH takes a string expression, rusqlite parameter binding is safe.

**FTS5 query syntax for user input:**
- User types `sunset mountains` → convert to FTS5 query `sunset OR mountains` for OR-based search
- OR provide the raw query to FTS5 MATCH (which defaults to AND-based matching)
- For Phase 7, pass the user input directly to MATCH — FTS5 handles tokenization. Phase 9 can add query preprocessing.

**Make all functions public (`pub fn`)** — they are called by the Tauri commands in Plan 05.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| File system → SQLite | SQLite database file read/write on local disk — no network boundary |
| FTS5 user query → SQLite | User search text passed to FTS5 MATCH via parameterized bind — SQL injection risk |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-007-04a | Tampering | SQL injection via FTS5 MATCH | mitigate | Use `params![query]` bind parameters, NOT string interpolation. rusqlite handles escaping for FTS5. |
| T-007-04b | Tampering | Migration transaction integrity | mitigate | rusqlite_migration wraps `to_latest()` in a transaction. SQLite rollback on crash. Per D-17. |
| T-007-04c | Information Disclosure | Database file permissions | mitigate | Connection opened with `PRAGMA synchronous=NORMAL` — balances durability and performance. Windows file permissions restrict to current user by default. |
| T-007-04d | Denial of Service | FTS5 query with large result set | mitigate | LIMIT/OFFSET pagination bounds result size. FTS5 BM25 ranking ensures most relevant results appear first. |
| T-007-04e | Elevation | SQLite user_version bypass | accept | rusqlite_migration uses SQLite's `PRAGMA user_version` for tracking. A malicious user with filesystem access could modify this, but in a local desktop app this is not a realistic threat vector. |
| T-007-SC | Tampering | Package legitimacy | accept | No new packages added — uses rusqlite and rusqlite_migration already in Cargo.toml from Plan 03. |
